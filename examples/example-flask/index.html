Defining the database connection

Instead of directly issuing SQL commands from our code, we'll use an object relational mapper (ORM), which will automatically store and retrieve your Python objects (users, posts, etc.) in the database. In particular, we'll use SQLAlchemy, a well known library of the Python ecosystem. Although we'll use it here to connect to MySQL or MariaDB servers, it is compatible with any other popular SQL database management system such as Oracle's.

First, we will import and configure the Flask-SQLAlchemy module in our application. Go to __init__.py and do the following:

    SQLAlchemy will use the PyMySQL library to connect to the MySQL or MariaDB database server.

Defining the database schema

We can now define the schema of the database. Replace the model.py module with this new version: 

import datetime
from typing import List, Optional

from sqlalchemy import String, DateTime, ForeignKey
from sqlalchemy.orm import Mapped, mapped_column, relationship
from sqlalchemy.sql import func

from . import db


class User(db.Model):
    id: Mapped[int] = mapped_column(primary_key=True)
    email: Mapped[str] = mapped_column(String(128), unique=True)
    name: Mapped[str] = mapped_column(String(64))
    password: Mapped[str] = mapped_column(String(256))
    posts: Mapped[List["Post"]] = relationship(back_populates="user")


class Post(db.Model):
    id: Mapped[int] = mapped_column(primary_key=True)
    user_id: Mapped[int] = mapped_column(ForeignKey("user.id"))
    user: Mapped["User"] = relationship(back_populates="posts")
    text: Mapped[str] = mapped_column(String(512))
    timestamp: Mapped[datetime.datetime] = mapped_column(
        DateTime(timezone=True), server_default=func.now()
    )
    response_to_id: Mapped[Optional[int]] = mapped_column(ForeignKey("post.id"))
    response_to: Mapped["Post"] = relationship(
        back_populates="responses", remote_side=[id]
    )
    responses: Mapped[List["Post"]] = relationship(
        back_populates="response_to", remote_side=[response_to_id]
    )
The User class defines the schema of the table that will store users in the database. Extending db.Model makes SQLAlchemy aware of this class as part of the database model.
Users in the database will have an integer identifier, which is the primary key of the table.
The column for email addresses will store strings up to 128 characters.
The email address of each user must be unique in the database.
The posts attribute of User objects will contain the list of posts (objects of the Post class) authored by that user. It's a one to many relationship between users and posts (one user can author many posts). The users table is the parent side of the relationship.
This attribute does not define a column in the database, but a relationship between the user and post tables. The other side of this relationship is defined in the user attribute of the Post class, as specified by the back_populates argument.
The Post class defines the schema of the table that will store posts in the database.
Since the posts table is the child side of the relationship with the users table, each post will have an integer column that stores the identifier of the user who authored it. This column is a foreign key to the id column of the users table.
The user attribute of Post objects will contain the User object that authored the post. It is, normally, more convenient to use this attribute than the user_id foreign key, which is just an integer. It's a many to one relationship between posts and users (many posts may be authored by one user). This attribute does not define a column in the database, but the other side of the relationship defined by the posts attribute of the User class.
The timestamp attribute of Post objects will contain the date and time when the post was created. Its data type is datetime.datetime, from Python's standard library.
The column for the timestamp attribute in the database will store timezone-aware date and time values.
If no value is provided for the timestamp attribute when a post is created, the column in the database will be set to the current date and time when the row is inserted into the database.
There is a many to one relationship between posts and themselves, as many posts may be responses to one post. A foreign key column needs to be declared for that relationship. The posts table is both the parent of the relationship (when considered from the point of view of the original post, the one side) and the child (when considered from the point of view of response posts, the many side). It is marked as Optional (it may take a None value in Python and a NULL value in the database) because original posts that are not responses to any other post will have no value for that attribute / column.
The response_to attribute of Post objects will contain the Post object to which this post is a response to (or None if it isn't a response). This attribute does not define a column in the database, but the child side of the relationship defined by the responses attribute of the same class.
Since the relationship is between posts and themselves, SQLAlchemy needs some help to understand which side of the relationship is which. The remote_side argument indicates that the id column is the one that defines the parent side of the relationship.
The responses attribute of Post objects will contain the list of Post objects that are responses to this post. This attribute does not define a column in the database, but the parent side of the relationship defined by the response_to attribute of the same class.
Since the relationship is between posts and themselves, SQLAlchemy needs some help to understand which side of the relationship is which. The remote_side argument indicates that the response_to_id column is the one that defines the child side of the relationship.

User and Post, apart from being normal Python classes, now define also the schema of the database, because they extend SQLAlchemy's Model class. They will be mapped to two tables, named user and post, in the database.

The columns and relationships of the two tables are defined by the attributes of the classes. The following syntactic elements are used to define them:

  • Mapped[...]: the data type of the attribute. There are examples, in the code above, of integers, strings, dates with time, objects of the classes User and Post, and lists of Post objects. When Optional is used, the column is allowed take a NULL value. Otherwise, the column is required to have a non-null value.
  • mapped_column(...): the column definition. Its arguments are used to specify other characteristics of the column, such as the maximum length of a string, whether the column is unique ot whether it is a primary or foreign key.
  • relationship(...): the definition of a relationship between tables. Relationships have two sides (for example, the posts attribute of User and the user attribute of Post). The back_populates argument refers to the name of the attribute in the other class that defines the other side of the relationship. Relationships from one table to itself are more complex to define, as you can see in the response_to and responses attributes: they also need the remote_side argument, which refers to the primary or foreign key that defines the other side of the relationship.

The User class defines a database table with the following characteristics:

  • Column id: an integer number, the primary key of the table.
  • Column email: a string, which must be unique and contain at most 128 characters.
  • Column name: a string, with 64 characters at most.
  • Column password: a string, with 256 characters at most.
  • The posts one to many relationship with the table that stores posts (one user can author many posts). The posts attribute of User objects will contain the list of all the posts (objects of the Post class) from this user.

The Post class defines a database table with the following characteristics:

  • Column id: an integer number, the primary key of the table.
  • Column user_id: post author, a foreign key to the users table.
  • The user many to one relationship with the table that stores users. It is the other side of the posts relationship defined in the User class. This relationship works through the user_id foreign key defined above.
  • Column text: a string with the text of the post.
  • Column timestamp: post creation date and time. When not specified, it defaults to the date and time when the row is inserted into the database.
  • Column response_to_id: a foreign key to the posts table itself. It defines to which post this one is a response, if any. If the post is not a response, the NULL value will be set here, which is allowed because of the Optional declaration.
  • The response_to many to one relationship with the same table (many posts may respond to one post). It defines a response_to attribute for Post objects, which provides the Post object to which this post is a response to (or None if it isn't a response). This relationship works through the response_to_id foreign key defined above.
  • The responses one to many relationship with the same table (one post may have many responses). It is the other side of the response_to relationship defined above. It defines a responses attribute for Post objects, which provides the list of Post objects that are responses to this post.

Registering new users

Although authentication and authorization seem at first glance the same thing, they aren't. Authentication consists in identifying users given the credentials they provided. These credentials frequently consist of an e-mail address and a password, although nowadays it's best practice to include also some other authentication factor or not using passwords at all.

Authorization consists in, given a resource and a user that has been previously authenticated, verifying whether that user is granted permission to access the resource. For example, to access the resource that deletes a particular post in a social network, only the user who created the post and certain administrators of the social network should be authorized.

The next step to build the application is managing user authentication, so that users can sign up, sign in and sign out. In this exercise we'll begin to work towards that goal by letting users create new accounts.

First, create a new template for the registration form and save it as templates/auth/signup.html. You can use the following example code or write your own: 

{% extends 'base.html' %}

{% block content %}
    <h2>Sign Up!!!</h2>
    <form action="{{ url_for('auth.signup_post') }}" method="post">
        <div><label>Email: <input type="email" name="email" required/></label></div>
        <div><label>User name: <input type="text" name="username" required/></label></div>
        <div><label>Password: <input type="password" name="password" required/></label></div>
        <div><label>Repeat password: <input type="password" name="password_repeat" required/></label></div>
        <div><input type="submit" value="Sign Up"/></div>
    </form>
{% endblock %}
Instead of repeating the boilerplate HTML code in every template, we extend the base.html template that already contains that code, and just define the content of the content block, which base.html expects.
The form will be submitted to the path of the signup function of the auth blueprint, which we'll define next.
The form will be submitted through a POST HTTP request. This is the appropriate method to use when the request modifies server state, as it will do when creating a new user.

The form will collect from the user four parameters, named email, username, password and password_repeat. When the user clicks on the Sign Up button, a POST HTTP request with those parameters will be sent to the path of the auth.signup_post controller, which we'll program next.

In order to keep our code modular, we'll create a new blueprint for code related to user authentication. Save the file microblog/auth.py with the following code: 

from flask import Blueprint, render_template, request, redirect, url_for, flash
from werkzeug.security import generate_password_hash

from . import db
from . import model

bp = Blueprint("auth", __name__)


@bp.route("/signup")
def signup():
    return render_template("auth/signup.html")


@bp.route("/signup", methods=["POST"])
def signup_post():
    email = request.form.get("email")
    username = request.form.get("username")
    password = request.form.get("password")
    # Check that passwords are equal
    if password != request.form.get("password_repeat"):
        return redirect(url_for("auth.signup"))
    # Check if the email is already at the database
    query = db.select(model.User).where(model.User.email == email)
    user = db.session.execute(query).scalar_one_or_none()
    if user:
        return redirect(url_for("auth.signup"))
    password_hash = generate_password_hash(password)
    new_user = model.User(email=email, name=username, password=password_hash)
    db.session.add(new_user)
    db.session.commit()
    return redirect(url_for("main.index"))
A new blueprint is created to hold all the code related to user authentication. The name of the blueprint is auth.
The signup function is registered to process requests to the /signup path of the application with a GET method (the default when no method is specified).
The signup function just renders the signup.html template that contains the registration form above.
The signup_post function is registered to process requests to the /signup path of the application with a POST method. These are the requests that will be sent from the form above.
The parameters sent from the form are accessed through the request.form dictionary-like object. The get method is used to get the value of each parameter given its name.
If there is an error in the data provided by the user, the user is redirected to the registration form again, i.e. the controller defined by the signup function above.
The query will run on the table that stores users.
The email column of the users table (model.User.email) is compared to the email address the has user entered in the form, which is stored in the email variable.
The query is executed through the db.session.execute method.
The result of the query is converted to a single User object or None if there are no results. Although it is impossible in this case because emails have been set as unique in the model, the method would raise an exception if there were more than one result.
Encrypt the password with the scrypt algorithm before storing it in the database.
Finally, when everything goes well, the user is redirected to the main page of the application.

You are defining two controllers for the /signup path. The first one will answer to GET requests for that path, since that's the default when you don't specify a method. It will just show the sign up form. The second one will answer to POST requests. It will get the parameters sent from the sign up form and actually create the user in the database if everything is correct.

In the second controller you can see how to access request parameters that come from a form, by calling the request.form.get method with the name of the parameter as argument. Note that parameter names are those you set as the value of the name attribute of each form control.

You can also see how to use SQLAlchemy through the Flask-SQLAlchemy extension to query the database for users given their email address:

  • First, a query is built with the db.select function. The query runs on the table for users and looks for users whose email is equal to the one provided.
  • Then, the query is executed with the db.session.execute function.
  • Finally, the scalar_one_or_none method returns the first result of the query as a User object, or None if there are no results. Although it is impossible in this case because emails have been set as unique in the model, the method would raise an exception if there were more than one result.

When everything goes well, a new User object is created and stored into the database:

  • New objects to be stored into the database are created with their constructor (in this case, the constructor for the User class).
  • Then, they are set to be added to the database with the db.session.add method.
  • Finally, the db.session.commit method is called to actually store the new object into the database. In general, changes to the database are not applied until this method is called.

The user is then redirected (HTTP redirection response) to the index page through the return redirect(...) statement. If the email already exists in the database or passwords are different, no user is created and the user is redirected again to the same form.

The scrypt algorithm is used to encrypt user passwords in the database, so that they are hard to crack even if the attackers get a copy of the database. This algorithm is included with the Flask framework. The encrypted password is created with the generate_password_hash function and stored in the database.

The auth blueprint you have just created in this exercise needs to be registered in the Flask application. To do so, you must edit the __init__.py module in the same way you have already done to import the main blueprint (that is, import the auth module and register it).

In order to check your code, run the Flask server and request the /signup path.

Providing feedback in case of user errors

When user registration fails no feedback is sent to the user, who won't know what went wrong. Flask provides the flash mechanism to pass posts from the context of one request to the next request from the same user session. You can flash the post at the controller and then show flashed posts at the form template.

In order to flash the post at the controller, add the following three calls to the flash function at auth.py: 

(...)
if password != request.form.get("password_repeat"):
    flash("Sorry, passwords are different")
    return redirect(url_for("auth.signup"))
(...)
if user:
    flash("Sorry, the email you provided is already registered")
    return redirect(url_for("auth.signup"))
(...)
flash("You've successfully signed up!")
return redirect(url_for("main.index"))
Flash messages are stored until they are consumed in a future request from the same user session.

In order to show flashed messages, add the following code to the proper place at your sign up form template: 

{% with messages = get_flashed_messages() %}
    {% for message in messages %}
        <div class="notification">{{ message }}</div>
    {% endfor %}
{% endwith %}
The get_flashed_messages function retrieves the list of flashed messages from the current user session that haven't been consumed yet.

Try this new version of the application and check that now you see the feedback in case of registration errors.

User authentication (II)

In the previous exercise we created the code to authenticate users, but we aren't remembering who has logged in yet. Instead of programming that part manually, we'll use the Flask-Login package, which automates session tracking for authenticated users, as well as access control to resources that need an authenticated user. You have already installed it in the previous lab.

First, activate the Flask-Login module at __init__.py. Import the module and then, somewhere within the create_app function, before registering blueprints, insert the following code: 

# With the other imports at the beginning:
from flask_login import LoginManager
(...)

# Inside create_app:
login_manager = LoginManager()
login_manager.login_view = 'auth.login'
login_manager.init_app(app)
from . import model

@login_manager.user_loader
def load_user(user_id):
    return db.session.get(model.User, int(user_id))
When the Flask-Login extension needs to redirect the user to the login form, it will redirect it to the path associated to the login controller function of the auth blueprint.
The function decorated with user_loader will be called by the Flask-Login extension when it needs to load a user object from the database given its id.
The model module needs to be imported because it's accessed in the load_user function. It's imported here, and not at the beginning of the file, to avoid circular imports, since the model module needs to import the db object from this module.
This is a convenient way of retrieving and object from the database given its class and its primary key. It's equivalent to composing a query that looks for a user whose id is equal to user_id, executing it and getting the first result.

That piece of code initializes the Flask-Login module, tells it that the view with the authentication form is auth.login and, finally, tells it how to load a user object from the database given its id.

Notice that the db.session.get method is used as a shortcut to retrieve an object from the database given its class and its id, without having to compose a query like the one in exercises 4 and 6.

In order to be compatible with the Flask-Login module, user objects need to inherit also from the class flask_login.UserMixin. Do this change at model.py: 

# With the other imports at the beginning
import flask_login
(...)

class User(flask_login.UserMixin, db.Model):
    (...)
The class extending UserMixin is the one that will be used by the Flask-Login extension to represent users.

In order to remember the user, go to the function at auth.py that authenticates users and make it communicate the identity of the user to the Flask-Login module: 

# With the other imports at the beginning
import flask_login
(...)

if user and check_password_hash(user.password, password):
    # The user exists and the password is correct
    flask_login.login_user(user)
    return redirect(url_for("main.index"))
(...)
The login_user function tells the Flask-Login extension that the user has been authenticated. From now on, the user will be remembered in future requests from the same user session.

You can get the identity of the authenticated user at templates through the current_user variable. It will contain its data if a user is currently authenticated: 

{% if current_user.is_authenticated: %}
    <div>User: {{current_user.name}}</div>
{% endif %}
The current_user variable is automatically made available to templates by the Flask-Login extension. Its value is the user object of the currently authenticated user, or an anonymous user object if no user is authenticated. Its is_authenticated attribute is True if a user is authenticated, and False otherwise.

At controllers, import current_user from flask_login and the current_user variable will also be available with the data of the currently authenticated user.

Modify base.html so that every page shows the name of the current user, if any.

Appendix 1: setting up a SQLite database to work in your own computer

Note: You are free to skip this section. This section isn't an official part of this lab. It's only provided for the convenience of students who want to work in their own computers locally.

Warning: Using a SQLite database makes it simple to work on these labs and the project locally in your own computer. However, you should be familiar with how to work with the MariaDB databases we provide, since it will be mandatory for your project submission to work with those databases. Moreover, SQLite databases aren't, in many cases, suitable for production web application environments.

If for some reason you want to work in your own computer, you can use a SQLite database instead of the MariaDB databases we provide. SQLite is a database management system that stores the database in a single file in your computer. You do not need to download or install anything else to use it, since it is included in Python's standard library.

In order to use SQLite, you need to change the database URI at __init__.py to the following: 

app.config["SQLALCHEMY_DATABASE_URI"] = "sqlite:///microblog.db"
This URI tells SQLAlchemy to use SQLite and to store the database in a file named microblog.db inside the instance subdirectory of your application. The instance subdirectory is created automatically by Flask when you run the application for the first time.

The database will be stored into the local file microblog.db, inside the subdirectory called instance that Flask will create.

Note that, after changing the database URI, you'll need to create the tables inside this new database with the db.create_all() method call, as explained in this piece of code.

The mysql client program won't work with SQLite but, if you want to explore the contents of your database, you can use specific software for SQLite such as DB Browser for SQLite. You just need to open the file instance/microblog.db with that program. You can also use the sqlite3 command line client to query and manage a SQLite database from a terminal.
